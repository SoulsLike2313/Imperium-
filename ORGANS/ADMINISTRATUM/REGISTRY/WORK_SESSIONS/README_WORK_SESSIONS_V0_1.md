# WORK_SESSIONS v0.1 (Administratum)

## Назначение
`WORK_SESSION` фиксирует длинную задачу Servitor в проверяемом формате, чтобы прогресс не терялся между этапами и сменами исполнителя.

## Почему это нужно
- Длинные Act 5 prefire задачи не должны начинаться заново после каждого короткого прогона.
- Нужен явный мост: отчёт этапа -> проверка -> ACK -> решение о продолжении.

## Ключевое правило
Servitor не может сам себе разрешить продолжение. Разрешение на продолжение оформляется только через `ADMINISTRATUM_ACK` и остаётся операционным решением, а не doctrinal/Owner authority.

## Как работает ACK
1. Servitor публикует `STAGE_PROGRESS_REPORT` с фактами и доказательствами.
2. Administratum проверяет отчёт и выпускает `ADMINISTRATUM_ACK`.
3. Решение в ACK: `CONTINUE_ALLOWED` / `STOP_OWNER_REQUIRED` / `BLOCKED` / `NEEDS_REVIEW`.

## Эскалация Owner
Если риск высокий, контекст неоднозначен или нужен policy-выбор, ACK должен требовать Owner-решение (`STOP_OWNER_REQUIRED` или `NEEDS_REVIEW`).

## Что это даёт для self-build задач
Скелет v0.1 даёт машинно-читаемый каркас для продолжения длинных задач без фейкового автозелёного статуса.

## Ограничение v0.1
Этот слой сам по себе НЕ делает Act 5 execution ready и НЕ меняет `READY_FOR_AGENT`.
