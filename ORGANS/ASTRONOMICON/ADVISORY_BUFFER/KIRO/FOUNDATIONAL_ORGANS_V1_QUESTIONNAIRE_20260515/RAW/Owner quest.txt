# 1. Executive Summary

## Why These Questions Matter Before Hardening

Hardening without clear organ definitions creates three failure modes:

1. **Scope Creep** — organs absorb functions that belong elsewhere, creating tangled dependencies
2. **Fake Green** — dashboards show green without real backend evidence, masking rot
3. **Boundary Collision** — two organs claim the same artifact type, causing ownership disputes

This questionnaire forces explicit decisions before any hardening script is written. Each answer becomes a constraint that shapes:
- what files/scripts each organ owns
- what checks prove the organ is alive
- what dashboard panels display real truth
- what gates block progression when evidence is missing

## How Answers Become a Hardening Plan

```
Owner Answers
    ↓
Decision Matrix (what each organ owns/rejects)
    ↓
V1 Criteria Checklist (testable pass/fail)
    ↓
Hardening Stages (ordered implementation phases)
    ↓
Scripts/Schemas/Checks (concrete artifacts)
    ↓
Dashboard Panels (visual truth layer)
    ↓
Test Plan (verification that V1 criteria are met)
    ↓
Registration Package (ORGAN_REGISTRY.json update)
    ↓
Execution Plan (who does what, when)
```
