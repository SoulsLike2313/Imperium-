# Owner Answer Sheet V0.1

Task ID: TASK-20260515-FOUNDATIONAL-ORGANS-V1-DEFINITION-V0_1

Status: Owner direction captured from live discussion.
Canonical language: English.
Source language: Owner spoke in Russian; this file stores the normalized English architecture meaning.

## 1. Administratum and Astronomicon split

Decision:
- Administratum owns task execution truth after a Servitor or Owner brings a TASK_ID.
- Astronomicon owns future task memory, task scope, stage maps, registration workflow, and planning memory.
- Administratum receives a task ticket, looks it up in Astronomicon, obtains scope/address information, and issues a route sheet.
- Administratum confirms that the Servitor is executing the correct task before task start becomes valid.

Implication:
- Administratum is the execution black box.
- Astronomicon is the task memory and navigation map.

## 2. Advisory versus canon

Decision:
- Raw advisory material belongs in Astronomicon advisory buffer.
- Canon/law/doctrine acceptance belongs to Doctrinarium, with Owner review.

## 3. Evidence and receipts

Decision:
- Use the most stable evidence/receipt format.
- Canonical machine evidence should be JSON.
- Human-readable summaries may exist as Markdown when useful.
- No PASS without evidence.

## 4. Transitions and receipts

Decision:
- Meaningful task/stage transitions require lightweight receipts.
- Logging must be useful and not create excessive trash.
- Important execution facts must be recorded.

## 5. Dashboards are mandatory

Decision:
- Each foundational organ must receive its own dashboard.
- Each dashboard must have integration ports for Sanctum.
- An organ without a dashboard is not a full organ.

## 6. Dashboard truth

Decision:
- Dashboard claims require evidence, source paths, links, and reports.
- If there is no proof that a dashboard is telling the truth, the dashboard is not telling the truth.
- No mock green.
- Every shown script must work.
- Every shown button must work or be clearly disabled.

## 7. Sanctum integration

Decision:
- Each organ keeps its own backend/report truth.
- Sanctum aggregates the powers of all organs as a central command bridge.
- Sanctum should behave like an "Infinity Gauntlet" that gathers organ powers without erasing their unique styles.

## 8. Officio Agentis authority

Decision:
- Nothing autonomous acts without Owner authorization at this stage.
- Officio Agentis provides Servitor/agent role scope, mode contracts, response contracts, restrictions, and proof that the agent is acting under the right laws/contracts.
- Automatic assignment is not trusted for V1 without Owner-gated workflow.

## 9. Doctrinarium task start meaning

Decision:
- Task start gate means the Servitor asks Doctrinarium whether organs are alive and laws can be obeyed before work starts.
- Doctrinarium must show organ health, law state, law storage addresses, and proof that laws are readable and systemically executed.

## 10. Dashboard actions

Decision:
- Dashboard action buttons require confirmation and receipt when they execute meaningful work.
- Button execution should be lightly logged.
- No excessive log trash.

## 11. Language policy

Decision:
- Canonical repo/machine artifacts remain English/UTF-8 safe.
- Every dashboard and UI application must support two languages: English and Russian.
- UI labels, tooltips, help panels, and presentation text should be localized through controlled i18n resources.

## 12. Visual identity

Decision:
- Each organ dashboard should carry the spirit of its organ.
- Sanctum should unify these different visual identities into one beautiful coherent flow.
- Sanctum should not repaint all organs into one flat style.
- Desired mood: Warhammer/Imperial spirit, sci-fi, Jarvis style, cosmic animation, smooth architecture, and 60 FPS target where feasible.

## 13. Retention

Decision:
- Evidence and receipts should be archived, not casually deleted.
- Retention should be monitored and justified.

## 14. Metrics

Decision:
- Every organ needs metrics.
- Statuses must have definitions explaining when and why the status applies.

## 15. Hardening target

Decision:
- Hardening target is a complete logged task corridor through the four foundational organs.
- Servitor begins with TASK_ID.
- Servitor first checks Doctrinarium and Administratum.
- Servitor then checks Officio Agentis and Astronomicon.
- Administratum confirms task start.
- Administratum records every stage.
- Administratum collects the final bundle/continuity pack.
- Astronomicon visualizes stage progress using Administratum's execution truth.
- Doctrinarium verifies organ/law readiness.
- Officio Agentis verifies role/contract compliance.
- Sanctum aggregates all dashboards and organ powers.

## Key visual stage states

- Completed stage: changed color and dim glow.
- Active stage: active pulse and animation.
- Future stage: white-platinum slow stellar glow.
- Blocked stage: severe warning marker with reason and evidence.
