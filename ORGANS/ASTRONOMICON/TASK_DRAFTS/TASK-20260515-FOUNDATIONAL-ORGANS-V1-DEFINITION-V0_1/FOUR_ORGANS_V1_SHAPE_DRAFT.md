# Four Foundational Organs V1 Shape Draft

Status: draft derived from Owner answers.

## Core V1 formula

An organ can be considered V1 only when it has:

1. backend truth;
2. real scripts/checks;
3. reports and receipts;
4. evidence links;
5. a self-report or health report;
6. a dashboard;
7. a Sanctum integration port;
8. no-fake-green protections;
9. English canonical machine artifacts;
10. bilingual UI resources.

## Task corridor overview

A valid task corridor should work as follows:

1. Servitor or Owner provides TASK_ID.
2. Servitor asks Doctrinarium whether organs are alive and laws can be obeyed.
3. Servitor goes to Administratum with TASK_ID.
4. Administratum looks up the task in Astronomicon.
5. Administratum issues address sheet and route sheet.
6. Servitor enters role and mode through Officio Agentis.
7. Servitor reads the task goal, scope, and stages in Astronomicon.
8. Servitor accepts the stage map.
9. Administratum verifies the Servitor is executing the correct task.
10. Administratum confirms task start.
11. Servitor executes stages.
12. Administratum records each stage.
13. Astronomicon visualizes stage progress from Administratum truth.
14. Administratum collects final bundle and continuity pack.

## Astronomicon V1

Purpose:
- future task memory;
- task scope and stage maps;
- registration workflow;
- review/export/import packs;
- task progress visualization.

Must own:
- task drafts;
- stage maps;
- task registration readiness;
- advisory buffer references;
- task goal/scope records;
- stage visualization model.

Must not own:
- lifecycle execution truth;
- final stage execution metrics;
- role contracts;
- law/canon acceptance;
- final task bundle collection.

Dashboard:
- task registration workspace;
- stage map visualization;
- review pack export/import;
- Speculum/Kiro advisory pack handling;
- active/future/completed stage visuals;
- reads progress from Administratum.

## Administratum V1

Purpose:
- execution truth;
- task lifecycle;
- route/address sheet;
- black-box stage logging;
- receipts;
- final bundle and continuity pack.

Must own:
- task lifecycle records;
- task start confirmation;
- stage execution ledger;
- address book;
- chronicle;
- continuity pack collection;
- local/Git/private context maps.

Must not own:
- task scope authoring;
- stage map design;
- law/canon interpretation;
- role contracts;
- audit verdicts.

Dashboard:
- Servitor arrival trace;
- TASK_ID lookup trace;
- route sheet issued;
- start confirmation;
- stage ledger;
- bundle builder;
- continuity pack builder;
- local/Git/private history map.

## Officio Agentis V1

Purpose:
- role/mode/agent contracts;
- agent scope;
- prompt/response contracts;
- stop conditions;
- proof that agents operate under correct contracts.

Must own:
- Servitor/Logos/Speculum/Advisor role contracts;
- mode definitions;
- permissions and restrictions;
- prompt trigger rules;
- response contracts;
- assignment validity checks;
- agent law/contract read proof.

Must not own:
- task lifecycle state;
- stage routing;
- law/canon acceptance;
- final audit verdicts;
- execution itself.

Dashboard:
- role contract registry;
- mode permission matrix;
- agent scope viewer;
- prompt trigger rules;
- stop condition panel;
- law/contract read evidence;
- trust/proof panel.

## Doctrinarium V1

Purpose:
- law/canon/doctrine authority;
- organ health gate;
- task-start legal/readiness gate;
- proof that laws are readable and systemically executed.

Must own:
- law registry;
- doctrine registry;
- law hierarchy;
- canon acceptance receipts;
- law integrity checks;
- organ health gate;
- task-start gate verdicts;
- violation records;
- disabled Inquisition hook queue.

Must not own:
- raw task planning memory;
- execution lifecycle;
- role contracts;
- tool implementation;
- audit verdicts;
- dashboard rendering engine.

Dashboard:
- organ health matrix;
- law registry state;
- law storage paths;
- law read receipts;
- law execution/compliance reports;
- task start gate verdicts;
- violations and exceptions;
- disabled Inquisition hook queue.
