# Стилевой контракт Owner

## Формат отчётов

- Компактные таблицы вместо длинных абзацев
- Полные пути к файлам
- Честные статусы (SCAFFOLD, MOCK, FAIL — нормально)
- Русский для объяснений, английский для кода и схем

## Что Owner НЕ хочет видеть

- Маркетинговый язык ("revolutionary", "cutting-edge", "best-ever")
- Длинные вступления перед сутью
- Скрытые проблемы за красивыми словами
- Автоматические действия без предупреждения
- Fake green в любой форме

## Что Owner хочет видеть

- Что сделано (конкретно, с путями)
- Что НЕ сделано (честно)
- Что mock/scaffold/not implemented
- Какой следующий шаг
- Где открыть результат
- Вердикт чекера

## Формат финального ответа

1. Название шага
2. Путь к результату
3. Вердикт
4. 3-4 строки комментария
5. Что открыть вручную
6. Что осталось mock/scaffold
7. Подтверждение что git scope не нарушен
