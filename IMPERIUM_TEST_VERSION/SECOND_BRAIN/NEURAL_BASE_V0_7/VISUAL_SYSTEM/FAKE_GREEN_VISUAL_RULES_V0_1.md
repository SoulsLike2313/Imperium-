# FAKE GREEN VISUAL RULES V0.1

## What Fake Green Means
Fake green is any visual implication of success/readiness/health without backend truth and receipt evidence.

## Green Color Restrictions
- Green cannot be default decorative color for global ambiance.
- Green is reserved for evidence-backed success states only.
- If receipt is stale/missing, green must be removed or downgraded.

## PASS Restrictions
- PASS labels require source receipt id and timestamp.
- PASS cannot be inferred from visual smoothness or animation quality.
- PASS must be blocked when required backend source is unavailable.

## Readiness Wording Restrictions
- "Ready" must specify what is ready (handoff/execution/preview).
- "Execution Ready" requires execution-path receipts.
- If only handoff exists, wording must remain handoff-only.

## Screenshot-Only Deception Risks
- Screenshots can hide stale/missing backend state.
- Screenshots cannot be sole evidence for green status.
- Every screenshot review must reference machine-readable receipts.

## Stale Data Visual Requirements
- Stale state must be explicit and persistent until refreshed.
- Stale labels must not be color-identical to fresh-success labels.
- Unknown data must never be auto-promoted to ready/green.

## Backend Unavailable Requirements
- Show degraded/unavailable state explicitly.
- Suppress success badges while backend is unavailable.
- Provide last-known timestamp and unverified marker.

## Performance Claim Requirements
- Any performance statement must include measurement receipt.
- No "fast/smooth/optimized" claim without numeric evidence.
- If measurement absent, visual must state "not measured".

## Task Handoff vs Execution Wording Rules
- "Handoff prepared" is not equal to "execution completed".
- "Execution complete" requires completion receipt.
- Mixed wording must default to safer, lower-claim phrasing.
